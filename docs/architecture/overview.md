# 架构概览

本项目的核心想法: 把程序直接编译到 transformer 的权重矩阵里, 让推理循环本身变成一台计算机。不是外部解释器, 不是工具调用。每一条指令的获取和每一次内存读取, 都通过抛物线注意力头完成。

这个项目的起源是独立验证 [Percepta 的声明](https://percepta.ai/blog/can-llms-be-computers): 2D 凸包注意力能否实现 O(log t) 的每步解码。经过 20 个研究阶段, 我们确认了 Percepta 的核心洞察是正确的, 但走了一条弯路才到达那里。

> [返回上级](../README.md) | [内存模型详解](memory-model.md) | [编译流程](compilation.md) | [ISA 参考](../isa/index.md)

## 项目目标

这个项目要回答一个具体的问题: **transformer 能不能变成一台通用计算机?**

不是"能不能模拟计算", 而是 transformer 的原生组件 (注意力头、前馈层) 能不能直接承担取指令、读写内存、执行运算的工作。答案是可以, 但有前提条件:

1. **注意力头负责寻址**, 前馈层负责运算和路由。两者分工明确。
2. **权重必须编译, 不能训练。** 梯度下降能学会执行的结构 (跳转、取指), 但学不会真正的整数加法。
3. **精度必须够高。** Float32 在抛物线编码下只能寻址约 4K 个位置; Float64 把范围扩展到 2500 万以上。

---

## 核心概念

### 抛物线注意力编码

项目使用的寻址机制是抛物线键编码。对于位置 `j`, 键向量设为:

```
k = (2j, -j²)
```

当查询向量 `q` 与这个键做点积时, 结果在目标位置产生尖锐的峰值。这比线性编码好得多: 线性编码的峰值会随距离增长而变钝, 抛物线编码在目标位置精确命中, 其他位置的响应迅速衰减。

所有五个内存空间共用同一套抛物线编码方案, 区别只在于查询向量的构造方式。

> 数学推导和寻址精度分析见 [内存模型详解](memory-model.md)。

### 编译执行, 非训练执行

这是本项目最重要的设计决策。20 个研究阶段中, 阶段 5-10 尝试了梯度下降训练, 结果:

| 能力 | 梯度下降 | 编译 |
|------|----------|------|
| 指令检索 (取指) | 能学会 | 直接写入 |
| 跳转路由 | 能学会 | 直接写入 |
| 数值翻倍 (a+a) | 加权损失下能学会 | 直接写入 |
| 真正的加法 (a+b, a≠b) | **学不会** | 直接写入 |

梯度下降在多任务上下文中无法学会真正的加法, 这是阶段 9 的核心发现。阶段 11 转向编译路径后, 100% 的执行轨迹正确, 包括算术、分支、函数调用和堆内存操作。

编译过程由 `_compile_weights()` 函数完成, 所有权重通过解析计算得到, 没有随机初始化, 没有反向传播。

> 编译流程的完整细节见 [编译流程](compilation.md)。

### Hard-max 注意力 (绝不使用 Softmax)

本项目使用 argmax 硬选择, 不使用 softmax 加权求和。原因很简单: 当键向量相同时, softmax 会给出均匀分布的权重, 这意味着注意力"不知道该看哪里"。Hard-max (argmax) 在任何情况下都做出明确的选择。

这不是一个可以妥协的设计选项。Softmax 在本架构下会失败, 这不是超参数调优能解决的问题。

### 五大内存空间

transformer 通过独立的注意力头组访问五个逻辑上分离的内存空间:

| 内存空间 | 寻址方式 | 用途 |
|----------|----------|------|
| **程序内存** | IP 寄存器指向当前指令 | 取操作码和参数 |
| **栈内存** | SP 寄存器指向栈顶, 同时读取 SP-1, SP-2 | 操作数栈, 压入/弹出值 |
| **局部变量** | 按索引号直接寻址 | LOCAL.GET/SET/TEE, 命名变量作用域 |
| **堆/线性内存** | 按地址寻址, 支持字节/短整/整型变体 | I32.LOAD/STORE, 数组和结构体 |
| **调用帧** | 帧指针寻址, 保存返回地址和 SP | CALL/RETURN, 递归调用 |

每个内存空间使用同一套抛物线编码 `k = (2j, -j²)`, 但查询向量的构造方式不同, 因此互不干扰。

## 关键设计决策

### 1. 编译, 不训练

阶段 5-10 的实验证明, 梯度下降可以学会执行的"结构" (取指、跳转路由), 但无法学会真正的算术。编译路径绕过了这个瓶颈, 直接将算术逻辑编码到前馈层的权重矩阵中。

这不是一个权衡。编译路径在所有维度上优于训练路径: 正确率从 85% 提升到 100%, 同时消除了训练所需的超参数调优、学习率调度和数据构造。

### 2. Float64 强制要求

抛物线键的点积分数按地址的平方增长。在 Float32 下, 当地址超过约 4000 时, 精度不足以区分相邻位置。Float64 把这个限制扩展到 2500 万以上, 对于目前的实验规模绰绰有余。

这不是一个可以事后优化的参数。如果用 Float32, 编译后的模型在中等大小的程序上就会产生错误的内存访问。

### 3. Hard-max 而非 Softmax

Softmax 在本架构下有根本性的问题: 当多个位置的键向量相似时, softmax 的输出趋近均匀分布, 注意力机制失去定位能力。Hard-max (argmax) 总是做出确定性的选择, 不管键向量有多接近。

### 4. 55 操作码的 WASM 风格 ISA

指令集以 WebAssembly 的 i32 子集为蓝本, 包含栈操作、算术、比较、位运算、控制流、局部变量、线性内存和函数调用。这不是一个最小化设计, 而是一个实用性设计: 足够丰富, 能编译真实的算法 (冒泡排序、递归斐波那契、匈牙利算法、数独求解器)。

> 完整的操作码定义见 [ISA 参考](../isa/index.md)。

### 5. 确定性执行

编译后的模型没有任何随机性。相同的程序和输入, 在 NumPy 后端和 PyTorch 后端上产生完全一致的执行轨迹。双后端验证是每个测试的标准流程。

## 模型规格

| 参数 | 值 | 说明 |
|------|-----|------|
| d_model | 36 | 模型维度 |
| head_dim | 2 | 每个注意力头的维度 |
| 操作码数 | 55 | WASM 风格栈机 ISA |
| 编译参数量 | ~964 | 全部通过解析计算得到 |
| 注意力机制 | Hard-max (argmax) | 禁止 softmax |
| 数值精度 | Float64 | 强制, 不可降级 |
| 内存空间 | 5 个 | 程序、栈、局部变量、堆、调用帧 |
| 寻址范围 (Float64) | 25M+ 位置 | 抛物线编码的理论上限 |
