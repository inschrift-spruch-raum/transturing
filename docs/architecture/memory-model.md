# 内存模型

这个项目的"内存"不是数组, 也不是哈希表。它是一组嵌入向量, 靠注意力机制来寻址。每次读内存就是一次点积加一次 argmax, 和 transformer 里的注意力头操作完全一样。

本文档详细说明抛物线编码的数学原理、五大内存空间的寻址方式、精度限制以及扩展机制。

## 抛物线键编码

### 核心思想

传统计算机用索引直接访问内存: `mem[3]` 直接定位到第 3 个位置。但在 transformer 里, 唯一的"寻址"机制就是注意力, 即 query 和 key 的点积。所以问题变成了: 怎么让点积恰好选中目标地址?

答案是抛物线编码。每个写入内存的值附带一个二维键:

```
key_j = (2j, -j²)
```

其中 `j` 是目标地址。读取时, 构造查询向量:

```
query_i = (i, 1)
```

点积得分函数:

```
score(j; i) = 2j·i + (-j²)·1 = 2ij - j² = -(j - i)² + i²
```

这是一个以 `j = i` 为顶点的开口向下的抛物线。当 `j = i` 时得分最高; 偏离目标地址时, 得分按距离的平方衰减。argmax 一定选中正确的条目, 不存在近似误差。

### 具体例子

`PUSH 3` 把值 `3` 写入栈地址 1, 生成键值对:

```
key = (2×1, -1²) = (2, -1)
value = 3
```

`PUSH 5` 把值 `5` 写入栈地址 2:

```
key = (2×2, -2²) = (4, -4)
value = 5
```

`ADD` 需要读取栈地址 2 和栈地址 1 的值。读取地址 2 时:

```
query = (2, 1)

score(地址1) = (2, -1) · (2, 1) = 2×2 + (-1)×1 = 3
score(地址2) = (4, -4) · (2, 1) = 4×2 + (-4)×1 = 4  ← 胜出
```

argmax 选中地址 2, 取到值 `5`。读取地址 1 时:

```
query = (1, 1)

score(地址1) = (2, -1) · (1, 1) = 2×1 + (-1)×1 = 1  ← 胜出
score(地址2) = (4, -4) · (1, 1) = 4×1 + (-4)×1 = 0
```

argmax 选中地址 1, 取到值 `3`。

### 代码实现

NumPy 执行器里的实现就是字面意义上的点积:

```python
def stack_read(addr):
    keys = np.array(stack_keys)       # 内存中所有二维键
    q = np.array([addr, 1.0])         # 查询向量
    scores = keys @ q                 # 点积
    best = np.argmax(scores)          # 硬注意力
    return stack_vals[best]
```

PyTorch 执行器更进一步, 使用真正的 `nn.Linear` 权重矩阵 (`W_Q`, `W_K`, `W_V`) 来实现抛物线编码。程序计数器、栈读取、参数获取和操作码分发全部通过 `matmul` 和 `argmax` 完成。

## 五大内存空间

所有内存空间使用完全相同的抛物线编码, 但各自有独立的注意力头。模型维度 `d_model=36`, 每个注意力头维度 `head_dim=2`。

### 1. 程序内存

**用途:** 取指令和取参数。

程序被存储为抛物线键值对序列。指令 0 的键是 `(0, 0)`, 指令 1 的键是 `(2, -1)`, 指令 2 的键是 `(4, -4)`, 依此类推。执行器根据当前程序计数器 (IP) 构造查询向量, 通过注意力获取当前指令的操作码和参数。

```
键编码:  key = (2 × ip, -ip²)
查询:    query = (ip, 1)
```

每一步执行都需要一次程序内存的注意力读取来获取下一条指令。

### 2. 栈内存

**用途:** 操作数栈的读取和写入。

栈是栈机 ISA 的核心工作区。每条算术指令从栈顶弹出操作数, 把结果压回栈上。执行器需要同时读取栈顶的三个位置:

- **SP:** 当前栈顶 (最后压入的值)
- **SP-1:** 栈顶下一个 (倒数第二个值)
- **SP-2:** 栈顶下两个

每个位置需要独立的注意力头。例如 `ADD` 从 SP-1 和 SP-2 读取两个操作数, 将结果写入 SP-1。

```
键编码:  key = (2 × stack_addr, -stack_addr²)
查询:    query = (stack_addr, 1)
```

栈指针 (SP) 通过累积和机制跟踪: 每一步记录 SP 的变化量 (PUSH 为 +1, POP 为 -1, 其他为 0), 累加得到当前 SP 值。累积和在 float32 下经过 10 万步也没有整数误差。

### 3. 局部变量

**用途:** 命名变量作用域, 对应 WASM 的 `local.get`/`local.set`/`local.tee`。

局部变量提供了比栈更直观的数据存储方式。你可以用 `LOCAL.SET 0` 把栈顶值存入局部变量 0, 用 `LOCAL.GET 0` 读回来。`LOCAL.TEE` 在写入的同时保留栈顶值不变。

```
键编码:  key = (2 × var_index, -var_index²)
查询:    query = (var_index, 1)
```

局部变量和栈内存使用不同的注意力头组, 地址空间互不干扰。这意味着局部变量 0 和栈地址 0 是完全不同的内存位置。

### 4. 堆/线性内存

**用途:** 任意地址的随机访问读写, 对应 WASM 的线性内存模型。

堆内存支持按字节、半字 (16 位) 和字 (32 位) 读写:

| 操作码 | 说明 |
|--------|------|
| `I32.LOAD` | 从堆地址加载 32 位值 |
| `I32.STORE` | 向堆地址存储 32 位值 |
| `I32.LOAD8_U` / `I32.LOAD8_S` | 加载字节, 零扩展/符号扩展 |
| `I32.LOAD16_U` / `I32.LOAD16_S` | 加载半字, 零扩展/符号扩展 |
| `I32.STORE8` | 存储低 8 位 |
| `I32.STORE16` | 存储低 16 位 |

```
键编码:  key = (2 × heap_addr, -heap_addr²)
查询:    query = (heap_addr, 1)
```

堆是五大内存空间中地址范围最大的。配合残差寻址 (见下文), 可以支持 25M+ 的地址空间, 足以实现数组、结构体等复杂数据结构。

### 5. 调用帧

**用途:** 函数调用时保存返回地址和栈指针。

`CALL addr` 执行时, 当前 IP+1 (返回地址) 和当前 SP 被压入调用帧。`RETURN` 执行时, 从调用帧中恢复这两个值。这支持了递归调用, 比如阶乘和斐波那契数列的递归实现。

```
键编码:  key = (2 × frame_depth, -frame_depth²)
查询:    query = (frame_depth, 1)
```

每次函数调用增加帧深度, 每次返回减少。嵌套调用通过帧深度的递增/递减自然地管理。

## 近期偏置 (Recency Bias)

### 问题: 写入覆盖

同一个地址可能被多次写入。比如栈地址 1 先被写入值 `3`, 后被写入值 `8`。在标准 hard-max 注意力下, 如果两个键完全相同, argmax 会选中得分并列的第一个, 不保证是最近写入的那次。

### 解决方案

给键的 y 分量添加一个极小的偏移量, 随写入次数递增:

```
key_j = (2j, -j² + ε × write_count)
```

其中 `ε` 是极小值 (NumPy 执行器用 `1e-10`, PyTorch 用 `1e-6`)。对同一地址的多次写入, 后写入的条目 y 分量略高, 在点积中得分也略高。这个偏移量足够小, 不会干扰不同地址之间的寻址精度。

经过测试, 近期偏置不会破坏其他地址的查询正确性。

## 精度限制

### Float32 的极限

抛物线编码的得分函数:

```
score(j; i) = -(j - i)² + i²
```

最大得分是 `i²` (当 `j = i` 时)。要区分目标地址和相邻错误地址 (得分差为 1), 需要:

```
1 / i² > machine_epsilon
```

即 `i < 1 / √ε`。

| 数据类型 | 机器精度 | 理论上限 (1/√ε) | 实测安全上限 |
|----------|----------|-----------------|-------------|
| float32 | 1.19×10⁻⁷ | ~2,896 | ~4,000 |
| float64 | 2.22×10⁻¹⁶ | ~67,108,864 | >50,000 |

float32 的安全上限约 4,000 个索引。NumPy 向量化操作的中间精度比最坏情况好一些, 早期测量在 ~7,283 才出错, 但保守估计安全上限是 ~4K。对于 WASM 级别的程序执行, 4K 内存单元非常受限。

### 为什么必须用 Float64

本项目**强制使用 float64**。原因:

1. 得分值按地址的平方增长, 地址 100 的得分是 10,000, 地址 4,000 的得分是 16,000,000
2. float32 只有 7 位有效十进制数字, 无法在如此大的得分基数上保持单位级的区分度
3. 堆内存和调用帧的地址可能远超 4K, float32 根本不够用

### 累积和的精度

栈指针的跟踪依赖累积和。即使使用 float32, 在 100,000 步内累积和的最大误差仅为 0.001, 没有整数误差。这是因为栈指针的变化量很小 (每次 ±1 或 ±2), 累积误差增长缓慢。所以累积和不是精度瓶颈, 抛物线编码才是。

## 残差寻址: 突破 Float32 限制

### 动机

虽然本项目用 float64 不受限制, 但 float32 的 4K 上限值得研究。Percepta 使用 float32 做推理, 他们必须有某种编码技巧来突破这个限制。残差寻址就是其中一种方案。

### 机制: 分块 + 双头

把地址 `addr` 拆成两部分:

```
block = addr // B
offset = addr % B
```

用两个注意力头分别寻址:

- **头 A:** 在块索引上做抛物线查找, 选出目标所在的块
- **头 B:** 在块内偏移上做抛物线查找, 选出目标条目

前馈层 (FFN) 组合两个头的输出。

### 扩展范围

每个头只需要处理 `B` 以内的地址。如果 `B = 5,000` (在 float32 安全范围内), 总地址空间为:

```
可寻址范围 = B × B = 5,000 × 5,000 = 25,000,000
```

只用 2 个注意力头就覆盖了 2,500 万个地址。压力测试验证了 330 个地址分散在 0 到 25M 范围内, 零错误。

### 对模型架构的影响

Percepta 的架构是 `d_model=36, 18 个注意力头`。分出 2 个头做块/偏移寻址后, 还剩 16 个头承担其他角色 (程序计数器、栈操作等)。这个分配在计算上是合理的。

## 总结

| 内存空间 | 寻址方式 | 注意力头数 | 用途 |
|----------|----------|-----------|------|
| 程序内存 | 抛物线 | 指令获取 | 取操作码和参数 |
| 栈内存 | 抛物线 | SP, SP-1, SP-2 | 操作数读写 |
| 局部变量 | 抛物线 | 变量索引 | 命名变量存取 |
| 堆/线性内存 | 抛物线 (+残差扩展) | 堆地址 | 随机访问读写 |
| 调用帧 | 抛物线 | 帧深度 | 函数调用/返回 |

所有内存空间共用同一套抛物线编码机制 `k = (2j, -j²)`, 区别只在于地址的语义不同。整个"内存系统"不是靠硬件电路, 而是靠注意力机制的数学性质实现的。

| 精度参数 | 值 |
|----------|-----|
| 编码维度 | 2D (head_dim=2) |
| 近期偏置 ε | 1×10⁻¹⁰ (NumPy) / 1×10⁻⁶ (PyTorch) |
| float32 安全上限 | ~4,000 索引 |
| float64 安全上限 | >67M 索引 |
| 残差寻址 (float32, B=5K) | 25M+ 索引, 2 个头 |
