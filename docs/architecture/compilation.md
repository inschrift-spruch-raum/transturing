# 编译过程：从训练失败到编译成功

本页解释本项目最核心的发现：为什么梯度下降无法让 transformer 学会执行程序，以及为什么将**解释器/执行器逻辑**直接编译进模型结构可以解决这个问题。要特别区分两层“编译”：一层是把通用执行器机制写进模型结构，另一层是把具体程序降成该执行器可运行的 ISA。

> 参见：[架构概览](overview.md) | [内存模型](memory-model.md) | [研究发现总结](../development/findings-summary.md)

## 核心结论

**编译，而非训练。** 将执行器的寻址与路由逻辑解析式写入模型结构后，transformer 就能正确执行程序；而具体程序本身则不是各自生成一套新模型参数，而是作为 ISA 程序输入执行器。

对当前代码库来说，程序导入路径已经收敛为**binary first**：`.wasm` 二进制模块先进入最小 binary frontend，再复用既有的结构化控制流映射与 lowering 语义，最终得到内部 `list[Instruction]`。

这一层支持范围仍然只覆盖当前已验证的 **i32-focused subset**。文档中的“支持 WebAssembly”应理解为这部分子集，而不是完整的通用 WASM 解码器。

## 训练路径（Phase 5-9）：一个有益的弯路

Phase 5 到 9 试图用标准的 next-token prediction + 交叉熵来训练 transformer 执行程序。这条路径精确地揭示了梯度下降在程序执行任务上的能力边界。

### Phase 5：初步训练

用 1000 个随机程序、最大 8 条指令来训练模型。结果：56% token 准确率（随机猜是 0.5%，所以模型确实学到了东西），但 0/50 条完整 trace 是正确的。

模型学会了执行的"语法"，但没学会"算术"。它知道什么时候该 PUSH、什么时候该 ADD，但算不对具体的数值。在自回归生成中，一个数字错了，后面全部跟着错。

宽度比深度重要。这印证了 Phase 4 的预测：attention head 的角色很清晰（取指令、读栈、跟踪 SP），真正的瓶颈在 FF 层的条件路由逻辑。

### Phase 6：课程学习

从简单指令（PUSH/HALT）开始，逐步增加难度。56% 提升到 85% token 准确率，完美 trace 从 0 增加到 39/50。

课程学习解决了**复制机制**的问题。模型学会了从栈上复制已有的值。但剩下的 11 条失败 trace 全部涉及真正的加法运算（a+b，a ≠ b）。

### Phase 7：架构变体

测试了 Percepta 的公开架构参数（d_model=36, 18 heads, 7 层）。结果：84.6% 准确率，DIFF+ADD 仍然是 0%。

无论是宽+浅还是窄+深，架构形状不影响结果。瓶颈不在模型结构上。

### Phase 8：瓶颈定位

这是整个项目最关键的诊断阶段。将每一步 trace 从 4 个 token 扩展到 6 个 token，显式展示两个操作数（FETCH1, FETCH2）。

结论极其清晰：

- **检索 100% 解决。** FETCH1 和 FETCH2 始终正确，模型完美地从不同栈位置取出两个操作数。
- **算术是唯一的瓶颈。** 唯一的错误发生在 `TOP = FETCH1 + FETCH2`。即使两个操作数已经在上下文中，模型仍然预测倍数（2a）或复制，而不是真正的 a+b。
- 模型**单独**学加法能达到 98% 准确率，但在多任务执行上下文中，加法只占约 15% 的训练信号。

### Phase 9：加权损失

将算术 token 的损失权重放大到 10-50 倍，测试是否纯粹是梯度信号不足的问题。

- 倍数运算（DUP+ADD, 2a）：83% → **100%**
- 相同操作数加法（SAME+ADD）：83% → **100%**
- **真正加法（DIFF+ADD, a≠b）：0% → 0%**。无论权重设多高都学不会。

**DIFF+ADD 墙不是梯度信号问题，而是表征问题。** FF 层无法在同时处理执行逻辑的情况下，在 token embedding 空间中学到整数加法。

### 训练瓶颈的完整递进

这五个阶段揭示了一条清晰的递进链路：

| 瓶颈 | 解决方式 | Phase |
|------|---------|-------|
| 复制机制 | 更多数据 + 课程学习 | 6 |
| 栈检索 | 微操作分解（micro-op） | 8 |
| 倍数运算（2a） | 加权损失 | 9 |
| 真正加法（a+b, a≠b） | **训练无法解决** | 5-9 |

每一步都以为"再努力一点就能突破"，但最终停在一堵真正的墙上。

## 转折点

Phase 9 的结果之后，我们重新审视 Percepta 的原始方法。发现 Phase 5-9 其实已经偏离了 Percepta 的路径：Percepta 是**编译**（analytically set weights），而我们是在**训练**（gradient descent）。训练路径很有启发性，但它不是正确的方法。

## 编译路径（Phase 11-13）：正确答案

### Phase 11：编译执行器

将注意力原语（抛物线索引 + 累积和）直接编译为 NumPy 运算。算术不再需要"学会"，而是直接编码进 FF 路由逻辑。

结果：**100% trace 匹配**，所有 Phase 4 的测试程序全部正确。包括真正的 a+b 加法。SUB、JZ/JNZ（条件跳转）工作正常，倒计时循环成为第一个成功执行的循环程序。

Phase 5-9 的 DIFF+ADD 墙是**训练**限制，不是**架构**限制。当执行器的关键寻址/路由机制通过解析方式固定下来后，transformer 可以正确执行程序。就当前代码实现而言，PyTorch 路径中的部分 opcode 语义仍由 Python 辅助逻辑完成，因此这里更准确的含义是“编译执行器”，而不是“每个程序都直接编译成一套独立权重”。

### Phase 12：真实 PyTorch 模型

将编译逻辑实现为真正的 `nn.Linear` 权重矩阵和 tensor 运算。

架构参数：
- `d_model=36`, `head_dim=2`
- 4 个活跃 attention head：程序操作码获取、程序参数获取、栈顶读取（SP）、次栈顶读取（SP-1）
- Hard-max attention（argmax，不是 softmax）
- FF 层分派：双线性门控（opcode one-hot × value routing matrix）
- Float64 精度（抛物线寻址正确性要求）

结果：**100% trace 匹配**，758 个编译参数。

### Phase 13：ISA 完备性

新增 SWAP、OVER、ROT 三个操作码。新增第 5 个 attention head 读取 SP-2。

编译后的 transformer 成为一台通用栈式计算机（等价于 Forth）：

- **Fibonacci(n)**：迭代计算，fib(10)=55，19 条指令，111 步
- **乘法(a,b)**：重复加法，mul(12,10)=120，119 步
- **2 的幂**：重复倍增，2^7=128，76 步
- **求和(1..n)**：累加循环，sum(1..15)=120，156 步
- **奇偶判断**：条件分支 + 反复减 2

全部在 NumPy 和 PyTorch 双执行器上 trace 级别匹配。

**最终参数：964 个编译参数，5 个 attention head，12 个操作码。**

## 权重矩阵结构

编译过程将解释器逻辑分解为四组权重矩阵/结构：

### W_Q, W_K（查询和键矩阵）

用于 attention head 的抛物线编码。`k = (2j, -j²)` 将位置 j 编码为一个 2D 抛物线点。点积注意力在目标位置产生尖锐峰值。

每个 attention head 负责：
1. **程序操作码获取**：从程序内存中取出当前 IP 处的操作码
2. **程序参数获取**：取出当前 IP+1 处的参数
3. **栈顶读取（SP）**：读取栈顶元素
4. **次栈顶读取（SP-1）**：读取栈顶下一个元素
5. **第三层读取（SP-2）**：为 ROT 指令读取

### W_V（值矩阵）

将 attention head 读到的原始值转换为模型可用的表示。包含地址校验逻辑：如果查询地址与存储地址不匹配，输出 0。

### FF 权重（前馈层）

核心的路由逻辑。采用双线性门控结构：

```
output = opcode_onehot @ W_route @ values
```

操作码的 one-hot 编码与路由矩阵相乘，选择对应的取值/分发逻辑。对当前仓库来说，这部分最准确的说法是：**寻址与分发机制**通过解析式方式固定下来；而在 PyTorch 后端里，部分 opcode 的具体算术语义仍由运行时辅助逻辑计算，而不是每个具体程序都重新生成一套模型参数。

## 三个核心洞察

## 程序导入路径

当前仓库里记录的程序导入路径是单一路径：

1. **binary WASM**
   - `.wasm` 二进制模块
   - 最小 binary frontend 解码为内部 Wasm 指令表示
   - 复用既有的结构化控制流映射与 ISA lowering
   - 产出 `list[Instruction]`

关键点不是发明另一套编译规则，而是让主入口直接面向真实工具链产物 `.wasm`，同时继续复用已经验证过的 lowering 逻辑。

### 当前支持边界

当前记录为已支持的范围是已验证的 i32 子集，包括：

- i32 常量与整数算术
- 比较与位运算
- 局部变量
- 结构化控制流
- 线性内存访问
- 函数调用

超出这部分范围的 Wasm 特性，不应被理解为当前文档已经承诺支持。

### 1. Attention 是查表，前馈层是路由和算术

2D 抛物线 attention 原语优雅、可组合、可靠地可学习。困难的始终是 FF 层中的条件逻辑。对于算术运算，必须编译，不能训练。

### 2. 训练弯路是意外但有价值的

这不是有意偏离 Percepta 的编译方法。但这段弯路精确地刻画了梯度下降在程序执行任务上的能力边界：检索可以学会，路由可以学会，倍数在辅助下可以学会，真正的加法在多任务上下文中无法学会。

### 3. 编译，而非训练

Percepta 的核心洞察是正确的。当执行器的关键机制通过解析方式固定下来时，transformer 就能正确执行程序。对本仓库而言，更准确的表述是：**通用执行器**被编译出来，**具体程序**再作为 ISA 输入该执行器。964 个参数，5 个 attention head，O(log t) 的每步复杂度，一台真正的计算机。

## 技术细节

- **Float64 强制要求**：抛物线寻址的得分值按 `addr²` 增长。Float32 在约 4K 索引时溢出。Float64 将范围扩展到 25M+。
- **Hard-max attention**：使用 argmax，绝不使用 softmax。当键相同时，softmax 会给出均匀权重，导致寻址失败。
- **i32 溢出语义**：所有算术运算应用 `result & 0xFFFFFFFF`（WebAssembly 标准）。
- **TRAP 错误处理**：除以零、栈下溢等运行时错误产生 OP_TRAP（操作码 99），不抛出 Python 异常。
